  2. 对分库分表不友好

drop truncate delete

turncate:删除表的数据，保留表的结构

delete:删除一行数据，如果不加 where 修饰，则删除全部数据

MySQL

开源成熟的关系型数据库，被广泛使用，默认端口3306

持久化存储数据

优点

  1. 成熟稳定，功能完善
  2. 开源免费
  3. 文档丰富，社区完善
  4. 对事务支持优秀
  5. 支持分库分表、读写分离、高可用
  6. 兼容各种操作系统，提供开箱即用的功能

MySQL 基础架构

  1. 连接器

    身份和权限验证

  2. 查询缓存

    优先查找缓存（新版本移除）

  3. 分析器

    词法、语法分析，看 SQL 要做什么

  4. 优化器

    按最优的方案去执行

    执行语句，从存储引擎拿到数据

  6. 插件式存储引擎

    负责数据的存储和读取，采用插件式架构，支持 InnoDB,MyISAM,Memory 等多种存储引擎

MySQL 存储引擎架构了解吗？

插件式架构，支持多种存储引擎

我们可以为不同的表设置不同的存储引擎来适应不同的场景

存储引擎是基于表的，而不是数据库

InnoDB MyISAM

InnoDB 是现在的默认存储引擎， MyISAM 是曾经的默认引擎

大部分情况选择 InnoDB 即可

区别

  1. 是否支持行级锁
  2. 是否支持MVCC
  3. 是否支持事务
  4. 是否支持外键
  5. 是否支持异常崩溃后的安全恢复
  6. 索引实现
  7. 性能

查询缓存

执行查询语句的时候，会先查询缓存

这个功能不太实用，已经被废除

查询缓存不命中

  1. 任意字符不同
  2. 包含函数，变量会导致缓存结果不储存
  3. 缓存涉及的表发生变化，缓存失效

性能提升 额外开销

索引

用于快速查找和检索数据

本质上是一种排序好的数据结构

优点

  1. 大大加快数据检索速度
  2. 创建唯一性索引，保证数据的唯一性

缺点

  1. 创建和维护索引耗费时间
  2. 索引占用物理空间

索引的底层数据结构

Hash 表

通过 hash 算法快速找到目标 value

缺点是不支持顺序和范围查询

B 树& B+树

主流索引 B+树

B 树& B+树两者有何异同呢？

  1. B 树所有节点都存放 key 和 data, 而 B+ 树只有叶子节点存放 key 和 data，非叶子节点只存放 key

  2. B 树的叶子节点都是独立的;B+树的叶子节点有一条引用链指向与它相邻的叶子节点。

  3. B 树查找可能没到叶子节点就找到了

    B+ 树检索效率稳定，任何查找都要到叶子节点

索引分类

实现

  1. B树索引
  2. 哈希索引
  3. 全文索引 文本分词搜索

存储

  1. 聚集索引

    数据和索引放在一起，主键作为索引

  2. 非聚集索引

    数据和索引分开

应用维度

  1. 主键索引
  2. 普通索引
  3. 唯一索引
  4. 联合索引
  5. 全文索引

主键索引

一张数据表有只能有一个主键，并且主键不能为 null，不能重复。

在 MySQL 的 InnoDB 的表中，当没有显示的指定表的主键时，InnoDB 会自动先检查表中是否有唯一索引且不允许存在 null 值的字段，如果有，则选择该字段为默认的主键，否则 InnoDB 将会自动创建一个 6Byte 的自增主键。

二级索引

又称辅助索引，它存储的数据是主键，通过二级索引定位到主键的位置

要经过两次查询

聚簇索引 非聚簇索引

聚簇索引：

索引与数据放在一起

InnoDB 中，一般用主键索引作为聚簇索引，叶节点直接存储数据

配合辅助索引使用，辅助索引中存储主键值

优点

  1. 查询速度快
  2. 顺序和范围查找

缺点

  1. 依赖有序的数据
  2. 更新代价大 所以主键一般都是不可修改的

非聚簇索引：

索引与数据分开

MyISAM 中，叶节点存储数据指针或者主键（辅助索引）

优点：

  1. 更新代价小

缺点：

  1. 依赖有序的数据
  2. 会二次查询

覆盖索引

查询的字段正好是索引的字段，那么直接根据该索引查到数据了，无需回表查询。

联合索引

使用表中的多个字段创建索引，就是 联合索引，也叫 组合索引 或 复合索引。

最左前缀匹配原则

在使用联合索引时，MySQL 会根据联合索引中的字段顺序，从左到右依次到查询条件中去匹配，依此过滤数据,直到匹配完成或遇到范围查询（<,>）

我们在使用联合索引时，可以把区分度高的放到左边。

索引下推

在非聚簇索引遍历过程中，对字段提前判断，过滤掉不符合条件的记录，减少回表次数。

选择合适的字段创建索引

  1. 不为 NULL 的字段
  2. 频繁查询的字段
  3. 作为查询条件的字段
  4. 频繁排序的字段
  5. 频繁链接的字段

索引实践

  1. 被频繁更新的字段应该慎重建立索引
  2. 限制每张表上的索引数量
  3. 尽可能的考虑建立联合索引而不是单列索引
  4. 注意避免冗余索引
  5. 删除长期未使用的索引

索引失效

  1. 使用 select * 查询
  2. 联合索引但不满足最左匹配原则
  3. % 开头的 LIKE

知道如何分析语句是否走索引查询

使用 EXPLAIN 对指令进行分析并显示结果

分析 SQL 的执行计划，并不会具体执行

MySQL 查询缓存

通过缓存提高效率，但是因为不实用已被废弃

语句完全相同触发缓存

缓存涉及到的表变更，缓存失效

查询中包含自定义函数，变量等，也不会缓存

数据库事务

保证对数据库的一系列操作构成逻辑上的整体，要么全部执行成功，要么都不执行

事务具有 ACID 特性

  1. 原子性
  2. 一致性
  3. 隔离性
  4. 持久性

并发事务带来了哪些问题?

  1. 脏读
  2. 丢失修改
  3. 不可重复读
  4. 幻读

不可重复读 幻读

不可重复读 记录的值发生改变

幻读 记录的个数发生改变

解决方式不一样

不可重复读： 记录锁 锁住数据即可

幻读： 临键锁 记录锁+间隙锁

并发事务的控制方式有哪些？

锁

读写锁

共享锁 排他锁

表级锁 行级锁

MVCC

多版本并发控制 一份数据存储多个版本，通过事务的可见性保证事务看到自己应该看到的版本

实现手段：隐藏字段,read view, undo log

事务隔离级别

  1. 读取未提交
  2. 读取已提交
  3. 可重复读
  4. 可串行化 最高隔离级别，所有事务依此执行，完全符合 ACID

表级锁 行级锁

表级锁：对整张表加锁，实现简单，粒度大，高并发下效率低

行级锁：对索引字段加锁，粒度小，高并发下表现好，但维护的开销大

一些行级锁

  1. 记录锁 对单个记录上锁
  2. 间隙锁 对范围上锁，不包括记录本身
  3. 临键锁 对范围和记录上锁，解决幻读问题

行锁默认使用的是 Next-Key Lock。但是，如果操作的索引是唯一索引或主键，InnoDB 会对 Next-Key Lock 进行优化，将其降级为 Record Lock

意向锁

意向共享锁（Intention Shared Lock，IS 锁）：事务有意向对表中的某些记录加共享锁（S 锁），加共享锁前必须先取得该表的 IS 锁。

意向排他锁（Intention Exclusive Lock，IX 锁）：事务有意向对表中的某些记录加排他锁（X 锁），加排他锁之前必须先取得该表的 IX 锁。

当前读 快照读

  1. 快照读

    一致性非锁定读，读取记录的历史版本，不用等待 X 锁的释放

    适用于对一致性要求不高但是追求极致性能的业务场景

    • 在 RC 级别下，对于快照数据，一致性非锁定读总是读取被锁定行的最新一份快照数据。
    • 在 RR 级别下，对于快照数据，一致性非锁定读总是读取本事务开始时的行数据版本。

  2. 当前读

    一致性锁定读，读的时候给记录加锁

MySQL 如何存储 IP 地址？

INET_ATON() ： 把 ip 转为无符号整型 (4-8 位)
INET_NTOA() :把整型的 ip 转为地址

如何分析 SQL 的性能？

我们可以使用 EXPLAIN 命令来分析 SQL 的 执行计划 。

执行计划是指一条 SQL 语句在经过 MySQL 查询优化器的优化会后，具体的执行方式。

Mysql 日志

包括错误日志，查询日志，归档日志，重做日志，回滚日志

比较重要的是 binlog,redolog,undolog

redo log

实现崩溃恢复能力

MySQL 实例挂了或宕机了，重启时，InnoDB存储引擎会使用redo log恢复数据，保证数据的持久性与完整性。

更新的数据记录到 Buffer Pool 中,以数据页为单位

具体操作写在 redo log buffer 中，刷盘到 redo log 中

redo log buffer -> page cache -> redo.file

innodb_flush_log_at_trx_commit

  • 0 事务提交不刷盘

    会丢失1秒的更改

  • 1 事务提交刷盘（默认）

    不会丢失

  • 2 事务提交将 redo log buffer 写入 page cache

    实例挂了不会丢失，宕机了会丢失1秒

重要的后台线程，每隔一秒将 redo log buffer 写入 page cache 然后 fsync 刷盘

日志文件组

环形数组

  • write pos 是当前记录的位置，一边写一边后移
  • checkpoint 是当前要擦除的位置，也是往后推移

redo log 可以提升效率

为什么不直接刷盘数据页？

数据页比较大，很可能只修改了几个字节，记录 redo log 并刷盘效率更高

记录 表空间号，数据页号，偏移量，更新值

binlog

redo log 物理日志，记录数据页上做了什么修改,属于存储引擎层

binlog 记录语句的原始逻辑，属于 Mysql Server 层

同步数据，保持数据一致性（集群之间）

binlog会记录所有涉及更新数据的逻辑操作，并且是顺序写。

格式

  • **statement ** 原始语句
  • row 解析后的数据
  • mixed 混合

写入机制

binlog cache -> page cache -> binlog

​ write fsync

参数sync_binlog

  • 0 每次提交事务时 write

    如果宕机了会丢失

  • 1 每次提交事务时 fsync

    不会丢失

  • N 提交N个事务时 fsync

    可能丢失 N 个事务，性能更好

undo log

记录事务中的操作，发生异常时根据 undo log 进行回滚

保证事务的原子性

回滚日志先于数据持久化到硬盘上，保证即使出现故障也能实现回滚

MVCC 的实现也依赖于 undo log

两阶段提交

防止出现 redo log 提交，binlog 没有提交的情况

将 redo log 的提交拆成 prepare，commit 两个阶段

其中 commit 在 binlog 写入后执行,保证两种日志的一致

一致性非锁定读 锁定读

一致性非锁定读

用版本号或时间戳标记版本，判断版本的可见性

读取数据时不会等待锁的释放，而是读取快照数据，也成为快照读

锁定读

对记录加锁，读取最新数据

InnoDB 对 MVCC 的实现

隐藏字段

  1. DB_TRX_ID 最后修改记录的事务 ID
  2. DB_ROLL_PTR 指向该记录的 undo log

ReadView

  1. m_low_limit_id 当前出现的最大事务 ID + 1
  2. m_up_limit_id 当前活跃的最小事务 ID
  3. m_ids 事务 ID 列表
  4. m_creator_trx_id 创建该 Read View 的事务 ID

undo-log

  1. 事务回滚时恢复数据
  2. 读取之前的版本数据，实现非锁定读

数据可见性算法

执行每个 select 语句前，都会创建一个快照（Read View）

  1. DB_TRX_ID < m_up_limit_id 修改该记录的事务在快照时已提交，记录对当前事务可见

  2. DB_TRX_ID >= m_low_limit_id 对该记录的修改在快照创建之后，记录对事务不可见

  3. m_up_limit_id <= DB_TRX_ID < m_low_limit_id

    在事务活跃列表中寻找 id，如果没有找到，说明在快照时已经提交，是可见的

    如果能找到，则在快照时未提交或快照之后才修改，数据不可见

  4. 当记录不可见时，读取记录的 DB_ROLL_PTR 指针所指向的 undo log 取出快照记录，重复以上步骤直到找到满足条件的版本。

RC 和 RR 隔离级别下 MVCC 的差异

在 RC 隔离级别下的 每次select 查询前都生成一个Read View (m_ids 列表)

在 RR 隔离级别下只在事务开始后 第一次select 数据前生成一个Read View（m_ids 列表）

MVCC➕Next-key-Lock 防止幻读

普通 select， MVCC 快照读
