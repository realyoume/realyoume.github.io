进程 线程

  1. 进程

进程是一个运行的程序

进程是操作系统进行资源分配和调度的基本单位

进程之间独立性高，便于管理资源

  1. 线程

线程是进程中的执行单位，一个进程可包括一个或多个线程

多个线程可以共享内存等资源

线程是轻量级的进程，线程的创建和切换开销小，性能优于进程，但安全性低

以 JVM JDK8 为例

一个 JVM 进程包含多个线程

线程共享 方法区，堆

线程私有 程序计数器，虚拟机栈，本地方法栈

并行 并发

并行：多个任务在同一时刻运行

并发：多个任务在同一时间段运行

同步 异步

同步：发出调用后阻塞，等待结果返回

异步：发出调用后不阻塞，之后通过通过或回调处理

多线程的优点

  1. 计算机底层

    充分利用多核 CPU，将多线程映射在多个核上同时执行，显著提高效率

  2. 互联网应用

    解决高并发问题，提高系统整体的并发能力和性能

多线程的缺点

  1. 共享资源的安全性
  2. 死锁
  3. 线程切换开销

线程的状态

NEW: 初始状态，线程被创建出来但没有被调用 start() 。

RUNNABLE: 运行状态，线程被调用了 start()等待运行的状态。

BLOCKED ：阻塞状态，需要等待锁释放。

WAITING：等待状态，表示该线程需要等待其他线程做出一些特定动作（通知或中断）。

TIME_WAITING：超时等待状态，可以在指定的时间后自行返回而不是像 WAITING 那样一直等待。

TERMINATED：终止状态，表示该线程已经运行完毕。

线程上下文切换

上下文：线程运行时的状态，如程序计数器，栈信息等

上下文切换：保存当前线程的上下文，将即将运行的线程的上下文加载进来。

频繁切换会造成系统效率下降

线程死锁

多个线程都在等待一个或多个资源的释放，由于线程被无限期的阻塞，所以程序不可能正常终止

四个必要条件

  1. 互斥

    资源只能被一个线程占用

  2. 请求并保存

    线程在请求资源而阻塞时，不会释放已获得的资源

  3. 非剥夺

    其他线程不能剥夺线程已获得的资源

  4. 循环等待

    多个资源对线程形成一种循环等待关系

预防和避免线程死锁

预防死锁

破坏死锁的必要条件

  1. 一次性申请所有资源
  2. 申请不到资源时可以主动释放获得的资源
  3. 按照顺序来申请资源，按照反序来释放资源

避免死锁

使用银行家算法进行评估，保证处于安全状态

安全状态：线程可以按照某种顺序依次完成

sleep() wait()

共同点：都可以让线程暂停

不同点：

  1. 锁

    sleep 不会释放锁

    wait 会释放锁

  2. 场景

    sleep 单纯用于暂停线程

    wait 用于线程间通信，与 notify,notifyAll 配合使用

  3. 唤醒

    sleep 预设时间后自动苏醒

    wait 要通过其他线程唤醒

  4. 对象

    sleep 是线程的方法

    wait 是对象的方法，因为要对对象的锁进行操作

start() run()

  1. start

    执行准备工作，调用 run 方法，真正的多线程执行

  2. run

    在 main 线程下单线程执行

volatile

保证可见性，不保证原子性

  1. 标识变量是共享且不稳定的，每次使用都在主存中读取
  2. 对变量的操作插入内存屏障，防止指令重排序

示例

双重校验锁实现对象单例

乐观锁 悲观锁

  • 悲观锁

    假设最坏的情况，认为每次访问都会出现问题，多写场景

    每次操作都会加锁，共享资源只能被一个线程使用

    synchronized ReentrantLock

  • 乐观锁

    假设最好的情况，每次访问不会出现问题，多读场景

    无需加锁，提交时检查是否被修改了，失败则重试

    版本号机制 CAS

乐观锁实现

版本号机制

用版本号记录数据修改的次数

只有版本号未改变时才成功，失败则重试直到成功

CAS 算法

Compare And Swap

用预期值和实际值进行比较，相等才会更新

CAS 是一个原子操作，底层依赖于一条 CPU 的原子指令

乐观锁问题

  1. ABA　问题

    发生了修改但检测不到

    解决方法，加入版本号和时间戳

  2. 循环重试开销大

    一直自旋重试带来很大开销

  3. 只能保证一个共享变量

synchronized

解决多线程访问共享资源的同步问题

保证被修饰的方法或代码块在任何时刻只有一个线程执行

  1. 修饰方法 锁实例对象
  2. 修饰静态方法 锁类
  3. 修饰代码块 锁指定对象、类

底层原理

  1. 代码块

    monitorenter monitorexit

    获得对象监视器 monitor

  2. 方法

    将方法标记为 ACC_SYNCHRONIZED，执行同步方法时，获得对象监视器 monitor

优化

偏向锁、轻量级锁、自旋锁、适应性自旋锁、锁消除、锁粗化等技术

四种状态：无锁状态、偏向锁状态、轻量级锁状态、重量级锁状态，随竞争激烈而逐渐升级

公平锁 非公平锁

  1. 公平锁

    先申请的线程先得到锁，为了保证公平，性能差

  2. 非公平锁

    按随机或优先级顺序获得锁，性能好，但会出现饿死现象

可中断锁 不可中断锁

  1. 可中断锁

    获得锁的过程中可以被中断，进行其他逻辑操作

    ReentrantLock

  2. 不可中断锁

    一旦申请了锁，只有拿到锁之后才能进行其他逻辑处理

    synchronized

ReentrantLock

实现了 Lock 接口，是一个可重入且独占式的锁，和 synchronized 关键字类似

底层实现依赖于 AQS

可重入锁

高级功能

  1. 可实现公平锁
  2. 等待可中断
  3. 选择性通知

ReentrantReadWriteLock

实现了 ReadWriteLock ，是一个可重入的读写锁

既可以保证多个线程同时读的效率，同时又可以保证有写入操作时的线程安全。

通过 AQS 实现

共享锁 互斥锁

StampedLock

性能更好的读写锁

Atomic 原子类

具有原子操作特征的类

保证线程安全

包括基本类型，数组类型，引用类型

ThreadLocal

保证每个线程都有自己的私有数据

线程会为访问的共享变量创建副本

Thread ThreadLocalMap ThreadLocal

数据存储在 ThreadLocalMap 中，ThreadLocal 封装了对 Map 的访问操作

ThreadLocalMap 懒加载，第一次 get set 才创建

内存泄露问题

ThreadLocalMap 中 key 为弱引用，value 为强引用

垃圾回收后出现 key 为 null 的 Entry，这时 value 永远无法被 GC 回收，产生内存泄漏

ThreadLocalMap 实现中已经考虑了这种情况，在调用 set()、get()、remove() 方法的时候，会清理掉 key 为 null 的记录。

使用完 ThreadLocal方法后 最好手动调用remove()方法

线程池

管理一系列线程的资源池

优点

  1. 降低资源消耗
  2. 提高响应速度
  3. 提高线程的可管理性

创建

  1. 通过ThreadPoolExecutor构造函数来创建

  2. 通过 Executor 框架的工具类 Executors 来创建

    • SingleThreadExecutor
    • FixedThreadPool
    • CachedThreadPool
    • ScheduledThreadPool

推荐线程都要通过线程池创建

通过ThreadPoolExecutor构造函数来创建,明确线程池运行规则，降低安全风险

重要参数

  1. corePoolSize 核心线程数
  2. maximumPoolSize 最大线程数
  3. workQueue 任务队列
  4. keepAliveTime 保存存活时间
  5. threadFactory 线程工厂
  6. handler 饱和策略

饱和策略

  1. DiscardPolicy 抛弃新任务
  2. DiscardOldestPolicy 抛弃最老任务
  3. AbortPolicy 抛出异常，拒接新任务的处理
  4. CallerRunsPolicy 调用自己的线程执行任务

阻塞队列

  1. 无界队列
  2. 同步队列
  3. 延迟阻塞队列

任务流程

线程命名

  1. 使用 ThreadFactoryBuilder setNameFormat
  2. 自己实现 ThreadFactory, setName

线程池大小

根据任务类型设置不同的线程池大小

  1. CPU 密集型任务

    核心数 + 1

  2. IO 密集型任务

    2 * 核心数

关闭线程池

shutdown（） :关闭线程池，线程池的状态变为 SHUTDOWN。线程池不再接受新任务了，但是队列里的任务得执行完毕。

shutdownNow（） :关闭线程池，线程的状态变为 STOP。线程池会终止当前正在运行的任务，并停止处理排队的任务并返回正在等待执行的 List。

Future

核心思想：将任务交给子线程异步执行，最后得到结果

FutureTask 类实现了 Future 接口，封装了 Callable,Runnable

我们可以通过ExecutorService.submit()，传入Callable或Runnable来得到 Future

CompletableFuture 更强大的 Future 实现，还提供了函数式编程，异步任务编排等任务。

AQS

AbstractQueuedSynchronizer 抽象队列同步器

用于构建锁和同步器

ReentrantLock,ReentrantReadWriteLock,Semaphore 基于 AQS　实现

实现原理

一套线程阻塞等待以及被唤醒时锁分配的机制

CLH队列锁，虚拟的双向队列

每个节点记录线程引用，线程状态，prev,next

state 表示加锁状态，初始值为 0

Semaphore

信号量，可以控制访问共享资源的线程数量

通过 AQS 实现，将 state 设置为 permit 即允许的数量

获得成功则 permit 减一

失败则加入阻塞队列

CountDownLatch

允许 count 个线程阻塞在一个地方，直至所有线程的任务都执行完毕。

CountDownLatch 是一次性的，计数器的值只能在构造方法中初始化一次

通过 AQS 实现，将 state 置为 count

线程使用 countDown() 用 CAS 的方式将 count 减一

使用 await()，如果 state 不等于0，则阻塞并不断重试

当 state 等于0时，执行之后的语句

CyclicBarrier

与 CountDownLatch 类似功能

基于 ReentrantLock 和 Condition 实现
