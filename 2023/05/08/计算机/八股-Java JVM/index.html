JVM

JVM 是 java 虚拟机，负责运行字节码，提供内存管理和垃圾回收等功能。

JVM 只是一种规范，可以有不同的实现，场景的有 HotSpot,J9

内存区域

程序计数器

一小块内存区域，不会溢出

记录字节码执行的行号

JVM 改变程序计数器来进行流程控制，分支、循环、条件等

线程切换时，通过程序计数器来恢复执行位置

线程私有

虚拟机栈

方法调用的数据需要通过栈进行传递，每一次方法调用都会有一个对应的栈帧被压入栈中，每一个方法调用结束后，都会有一个栈帧被弹出。

栈帧

  1. 局部变量表

    记录基本数据类型和对象引用

  2. 操作数栈

    记录中间计算结果和临时变量

  3. 动态链接

    用于方法调用时，把符号引用变成直接引用

  4. 返回地址信息

方法返回和抛出异常都会导致栈帧弹出

线程私有，避免其他线程访问本线程的局部变量

可能会出现 StackOverFlowError, 栈深度超过允许的最大深度

本地方法栈

作用与虚拟机栈类似

不同是虚拟机站为 Java 方法服务

本地方法栈为 Native 方法服务

堆

最大的一块内存区域，所有线程共享

主要是为了存放对象实例，几乎所有的对象和数组都在堆上分配内存（栈上分配，逃逸分析）

堆是垃圾回收的主要区域，也被成为 GC 堆

分为新生代，老年代，永久代（JDK1.7）

方法区

各线程共享

存放 class 文件相关信息

被虚拟机加载的类信息，字段信息，方法信息，常量，静态变量，被即时编译器编译后的代码缓存

实现

JDK1.7 永久代

JDK1.8 元空间

为什么用元空间代替永久代？

  1. 更大的空间，避免OOM
  2. 合并HotSpot和JRockit的代码

运行时常量池

存放常量池表加载后的内容

字面量，符号引用

字符串常量池

存放字符串常量，避免重复创建

StringTable 实现

位于堆中，享受垃圾回收

new 创建的字符串不会在常量池中

直接内存

在本地内存上分配，不会受到堆的大小限制，但也会 OOM

NIO，基于通道和缓冲区，使用 Native 函数库分配堆外内存，使用一个DirectByteBuffer对象作为引用进行操作

在一些场景显著提高性能，避免在 Native 堆和 Java 堆之间来回复制

StringBuilder 创建的字符串不会在常量池中

本地内存

通过本地方法分配的内存

不受 Java 堆大小限制，不会被内存管理

与虚拟机内存相对应

对象创建

类加载检测

通过符号引用找到对应的类，检查该类是否被加载过，如果没有就先加载

分配内存

将一块内存从堆中划分出来

  1. 指针碰撞

    适用于内存规整的场景，将指针移动一段空间即可

  2. 空闲列表

    适用于内存不规整的场景，会有一个列表记录空闲的空间，分配一块足够大的空间即可

如何解决并发问题？

  1. CAS+失败重试

    乐观锁，保证更新操作的原子性

  2. TLAB

    为每个线程预先在 Eden 区中分配一块空间，优先在 TLAB 中分配

初始化零值

将分配的内存区域全部置为0（除了对象头）

保证实例字段可以不赋值直接访问，可以访问到对应的零值

设置对象头

基础的设置

对象头包括类的元信息，哈希码，GC分代年龄等

执行 init 方法

把对象按照程序员的意愿进行初始化

对象的内存布局

  1. 对象头

    哈希码，GC分代，类的元数据

  2. 实例数据

    真实有效的信息

  3. 填充

    起占位作用，保证对象的大小是8字节的整数倍（HotSpot）

对象访问定位

  1. 直接指针

    直接通过内存地址访问

  2. 句柄

    根据句柄地址访问句柄池中的句柄，再根据句柄访问对象

使用直接指针的好处是速度快

使用句柄的好处是对象修改时只用修改句柄中的地址

内存分配和回收原则

  1. 对象优先在 Eden 区分配

    Eden 区满时，会进行 Minor GC

  2. 大对象直接进入老年代

    避免为大对象分配内存时由于分配担保机制带来的复制而降低效率

  3. 长期存活的对象将进入老年代

    每经过一次 Minor GC 年龄加一， Eden -> Survivor -> 老年代

    Survivor 年龄上限取默认值和一半大小的年龄的最小值

    Parnew 15, CMS 6

  4. 空间分配担保

    在进行 Minor GC 之前，确保老年代的空间足够容纳新生代的所有对象

    JDK 6 Update 24 之前 HandlePromotionFailure

    允许担保失败且老年代空间大于历次晋升的平均水平，进行 Minor GC，否则进行 Full GC

    JDK 6 Update 24 之后

    只要能空间容纳全部对象或大于历次晋升的平均水平，进行 Minor GC

死亡对象判断方法

引用计数法

记录对象被引用的次数，为0则说明对象死亡

简单易实现，但是无法解决循环引用的问题

可达性分析算法

从 GC Roots 对象作为起点进行搜索，走过的路径称为引用链，如果一个对象到任何 GC Roots 对象都没有引用链，说明该对象不可到达，即死亡。

可以作为 GC Roots 的对象？

  • 虚拟机栈(栈帧中的本地变量表)中引用的对象
  • 本地方法栈(Native 方法)中引用的对象
  • 方法区中类静态属性引用的对象
  • 方法区中常量引用的对象
  • 所有被同步锁持有的对象

对象可以被回收，就代表一定会被回收吗？

两次标记过程，如果有 finalize 方法就进入队列进行二次标记，如果还没有建立联系则回收

引用类型

强引用

最普遍的引用，绝对不会被回收，即使内存不足

软引用

可有可无的引用，内存足够时不会被回收，内存不足时被回收

弱引用

可有可无的引用，一旦被垃圾回收器发现，就会被回收

但是垃圾回收线程是一个优先级很低的线程，不一定能很快回收

虚引用

用来跟踪对象被垃圾回收的活动

垃圾收集器在准备回收引用时，会将引用加入引用队列，以此来通知程序该引用将会被回收，程序进行相应的处理操作

主要使用强引用和软引用，软引用可以防止 OOM

如何判断一个常量是废弃常量？

如果没有任何引用指向该常量，说明该常量是废弃常量

如何判断一个类是无用的类？

  1. 该类的所有实例都被回收
  2. 该类的类加载器被回收
  3. 该类的类对象不能在任何地方被引用，也不能通过反射方法访问到

垃圾回收算法

标记-清除算法

分为标记和清除两个阶段

先标记存活的对象，然后清理掉没有标记的

存在问题

  1. 效率问题
  2. 空间问题：会留下很多的不连续的空间碎片

复制算法

将可用内存分为大小相等的两块，每次使用其中的一块

清理时将存活对象复制到另一块去，然后一次性清理空间

存在问题

  1. 可用空间变小
  2. 不适用于老年代：大对象，复制效率低

标记-整理算法

适用于老年代

先标记存活对象，然后将对象向一边移动，清理掉边界以外的空间

分代收集算法

根据年龄代的特点选择不同的收集算法

比如

新生代 标记-复制 每次有大量对象死去

老年代 标记-整理 大对象，存活率高

垃圾收集器

Serial 收集器

新生代收集器，采用标记-复制算法

单线程垃圾收集器

垃圾回收时会暂停用户线程 stop the world

简单高效，单核条件下效率高

ParNew 收集器

新生代收集器，采用标记-复制算法

多线程垃圾收集器 相当于 Serial 收集器的多线程版本

垃圾回收时会暂停用户线程 stop the world

在多核条件下表现好

Parallel Scavenge 收集器

新生代收集器，采用标记-复制算法

多线程垃圾收集器

不关注停顿时间，关注吞吐量（用户代码运行时间占总时间的百分比）

停顿时间短的垃圾收集器适合与用户交互频繁的程序，较少的停顿能够提高用户体验

吞吐量大的垃圾收集器可以高效率利用CPU，主要适用于后台运算而不需要太多交互的内容。

Serial Old 收集器

Serial收集器的老年代版本,标记整理算法

单线程的收集器

Parallel Old 收集器

Parallel Scavenge收集器的老年代版本

使用的是标记整理算法

CMS收集器

老年代收集器，以最短停顿时间为目标，采用标记清除算法

流程

  1. 初始标记

    单线程执行，标记与 GC Roots 直接关连的对象

  2. 并发标记

    并发执行，从 GC Roots 对象开始追踪并标记对象

  3. 重新标记

    单线程执行，对并发标记阶段发现变化的对象进行标记修正

  4. 并发清除

    并发执行，清除之前标记的垃圾

存在问题

  1. 对 CPU 资源敏感
  2. 无法处理浮动垃圾
  3. 产生内存碎片

G1 收集器

面向服务器的垃圾收集器，多核大内存场景

不再分为三个年龄代，内存划分为等大的独立区域，每个区域可以有不同的年龄

特点

  1. 并行与并发

    充分利用多核缩短停顿的时间，部分垃圾回收操作与用户线程并行

  2. 空间整合

    从整体上来看是整理，从局部上来看是复制

    从一个或多个Region复制存活对象到堆上的另一个空的Region，并且在此过程中压缩和释放内存

  3. 可预测的停顿

    使用者指定在一个长度为M毫秒的时间片段内，消耗在垃圾收集上的时间不得超过N毫秒

    跟踪 Region 中垃圾堆积的价值大小，维护一个优先队列，根据运行回收的时间，优先回收价值最大的 Region,提高效率

流程

  1. 初始标记
  2. 并发标记
  3. 最终标记
  4. 筛选回收

优先回收价值高的 Region,只有并发标记阶段是并行的

Client Server

JVM 的两种模式

client:

轻量级模式，启动速度快，长期运行效率不如 server 模式

使用C1的轻量级编译器

Serial + Serial Old

server:

重量级模式，启动速度满，但长期运行效率快

C2的轻量级编译器

CMS + Parnew

默认的垃圾收集器也会有不同

类加载过程

加载 验证 准备 解析 初始化

连接：验证 准备 解析

加载

  1. 通过全类名获得二进制字节流
  2. 将静态存储结构到运行时数据
  3. 生成一个 class 对象，作为对方法区数据的访问入口

验证

确保 class 文件的信息符合《Java 虚拟机规范》的要求，保证这些信息被当作代码运行时不会危害虚拟机自身安全

  1. 文件格式验证
  2. 元数据验证
  3. 字节码验证
  4. 符号引用验证

准备

为类变量（即静态变量）分配内存并设置类变量初始值

初始值就是默认零值，引用类型是 null

解析

将符号类型转换为引用类型

主要是类，接口，方法，字段

初始化

执行 clinit 方法

clinit 方法根据静态变量和静态方法块自动生成，为静态变量赋值

Class 对象

  1. 和对象一样存放在堆中
  2. 只能由 JVM 创建和加载
  3. 作用是提供关于某个对象的类型信息，对反射很重要

类加载器

主要功能是将字节码加载到 JVM 的方法区中并创建 class 对象作为访问入口。

每一个类都有一个类加载器

需要时才去加载，对内存更友好

每一个类加载器都会记录已经加载的类，避免重复加载

内置 类加载器

  1. 启动类加载器

    顶层加载器，所有类加载器的父类

    负责加载核心类库下面的类

  2. 扩展类加载器

    负责加载 ext 目录下的所有类

  3. 应用程序类加载器

    负责加载用户的类

  4. 自定义类加载器

双亲委派模型

解决了类加载器的选择问题

流程

  1. 先判断类是否被加载过，是则直接返回
  2. 类加载器优先委托父类加载器进行加载（loadClass），于是所有的加载都会到顶层的 启动类加载器
  3. 只有当父类加载器无法加载时（它的搜索范围中没有找到所需的类），才会让子类加载器进行加载（findClass）

好处

  1. 避免类重复加载
  2. 防止核心 API 被篡改

打破双亲委派模型？

重写 loadClass 方法

判断两个类是否相同

  1. 全类名相同
  2. 类加载器相同

JVM 重要参数

命令行 含义
-Xmx 最大堆内存
-Xms 最小堆内存
-XX:NewSize 新生代初始内存
-XX:MaxNewSize 新生代最大内存

-XX:+UseParallelGC
-XX:+UseParNewGC
-XX:+UseG1GC G1 垃圾收集器
-XX:+HeapDumpOnOutOfMemoryError 内存溢出时将heap 转储到物理文件
-XX:HeapDumpPath 转储文件路径
-server 启用 Server Hotspot VM
