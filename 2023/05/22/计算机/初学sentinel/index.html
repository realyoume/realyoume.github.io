Sentinel

Sentinel 是阿里巴巴开源的一款微服务流量控制组件

雪崩问题

微服务雪崩是指在微服务架构中，由于某个微服务的不可用或延迟，导致依赖该微服务的其他微服务也不可用或延迟，最终导致整个系统不可用的现象。

核心：避免 tomcat 的连接数超过上限

处理方式

  1. 超时处理：超时后终止
  2. 舱壁模式：限制每个业务的线程数
  3. 熔断降级：通过断路器统计异常业务的比例，超过阈值就熔断该业务
  4. 流量控制：限制业务的 QPS（预防雪崩）

限流规则

簇点链路

项目内的调用链路，链路中的每个接口就是一个资源

流控、熔断都是针对簇点链路中的资源来设置的

流量控制

JMeter

访问流量测试工具

通过 UI 进行配置

流控模式

直接

统计当前资源的请求，触发阈值时对当前资源直接限流，也是默认的模式

关联

统计与当前资源相关的另一个资源，触发阈值时，对当前资源限流

两个有竞争关系的业务，比如访问相同资源

高优先级业务触发阈值时，对优先级低的业务进行限流

链路

统计从指定链路访问到本资源的请求，触发阈值时，对指定链路限流

将方法声明为 Sentinel 管理的资源

流控效果

到达阈值时采取的措施

快速失败

到达阈值后，新的请求立即拒绝并抛出异常。是默认的处理方式

预热模式

同样拒绝超出阈值的请求，但是会有一个阈值逐渐变大的过程

应对服务冷启动

threshold 最大阈值

预热时长

派对等待

所有请求进入队列，按照阈值允许的时间间隔依此执行

如果预期等待的时长超过最大时长，则会被拒绝

具有流量整形效果

热点参数限流

常规：统计某个资源的所有请求

热点参数限流：根据参数值的不同，分别统计请求

可以根据参数进行不同的限流，比如热点参数的阈值大一些

更细粒度的限流

热点参数限流对默认的SpringMVC资源无效

隔离与降级

虽然限流可以尽量避免因高并发而引起的服务故障，但服务还会因为其它原因而故障。而要将这些故障控制在一定范围，避免雪崩，就要靠线程隔离（舱壁模式）和熔断降级手段了。

服务的提供方已经宕机，核心是保护客户端（服务调用方），防止发生雪崩

线程隔离

限制对服务访问的线程数

  • 线程池隔离

    对不同服务创建不同线程池

    通过新线程来进行远程调用

    支持主动超时，异步调用

    线程的额外开销大

    场景：低扇出

  • 信号量隔离

    使用原有线程进行调用

    用信号量控制访问数量

    轻量级，无额外开销

    不支持主动超时，异步调用

    场景：高扇出，高频调用

熔断降级

断路器统计服务调用的异常比例、慢调用比例

超出阈值就熔断服务，拦截对该服务的一切请求

服务恢复时，放行请求

熔断策略

  1. 慢调用

    响应时长大于指定时长即为慢调用

    慢调用比例超过阈值，触发熔断

    参数：最大RT，比例阈值，熔断时长，最小请求数，统计时长

  2. 异常比例

    异常比例超过阈值，触发熔断

    参数：比例阈值，熔断时长，最小请求数，统计时长

  3. 异常数

    异常数超过阈值，触发熔断

    参数：异常数，熔断时长，最小请求数，统计时长
