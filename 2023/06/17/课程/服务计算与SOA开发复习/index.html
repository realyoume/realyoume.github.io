知识点

XML

XML（可扩展标记语言）是一种用于表示和存储数据的标记语言。它使用自定义标签来描述数据的结构和内容。XML的目标是提供一种通用的数据格式，以便不同的系统和应用程序可以交换和共享数据。

XSL（可扩展样式表语言）是一种用于对XML文档进行转换和处理的样式表语言。XSL提供了一种将XML文档转换为其他格式（如HTML、PDF等）或应用样式（如添加颜色、样式等）的机制。XSL使用XPath来定位和选择XML文档中的特定元素，并使用模板和规则来定义转换的方式。

XSL样式表使用XML语法编写

XML处理

SAX解析 Simple API for XML

逐行扫描文档，一边扫描一边解析。相比于DOM，SAX可以在解析文档的任意时刻停止解析解析，是一种速度更快，更高效的方法。

优点：不用事先调入整个文档，占用资源少。解析可以立即开始，速度快，没有内存压力。

缺点：不能对结点做修改

适用：读取XML文件

DOM解析 Document Object Model

为解析XML文档定义了一组接口。解析器读入整个文档，然后在内存中建立一个树结构， 然后就可以使用 DOM 接口来操作这个树结构。

优点：整个文档树在内存中，便于操作；支持删除、修改、重新排列等多种功能

缺点：如果文件比较大，内存有压力，解析的时间会比较长。将整个文档调入内存（包括无用的节点），浪费时间和空间。

适用：修改XML数据

SOAP

SOAP 是基于 XML 的简易协议，可使应用程序在 HTTP 之上进行信息交换。

一条 SOAP 消息就是一个普通的 XML 文档，包含下列元素：

  • 必需的 Envelope 元素，可把此 XML 文档标识为一条 SOAP 消息
  • 可选的 Header 元素，包含头部信息
  • 必需的 Body 元素，包含所有的调用和响应信息
  • 可选的 Fault 元素，提供有关在处理此消息所发生错误的信息

In SOAP processing model, two types of intermediaries are defined: forwardingintermediaries and active intermediaries

Message Exchange Protocal

Network Protocal

WSDL

WSDL 指网络服务描述语言 (Web Services Description Language)。

WSDL 是一种使用 XML 编写的文档。这种文档可描述某个 Web service。它可规定服务的位置，以及此服务提供的操作（或方法）。

元素 定义
web service 执行的操作
web service 使用的消息
web service 使用的数据类型
web service 使用的通信协议

定义方法时，不支持方法名相同的重载。

REST

符合 REST 标准的 Web 服务允许请求系统使用一组统一且预定义的无状态操作来访问和操作 Web 资源的文本表示

  1. REST 服务是无状态的
  2. REST 服务具有统一的接口
  3. 资源是通过表示来操作的
  4. 消息是自我描述的
  5. RESTful 架构是用资源构建的
  6. 超媒体作为应用程序状态的引擎
  7. REST API 不应依赖于任何单一的通信协议
  8. REST API 应该花费几乎所有的描述性工作来定义用于表示资源和驱动应用程序状态的媒体类型
  9. REST API 不能定义一成不变的固定资源名称，因为这样做会导致客户端和服务器的耦合
  10. REST API 应该有一个稳定的初始入口点 URI（可收藏）

优势：

  1. 额外的水平扩展能力
  2. 一种简单明了的缓存机制
  3. 一个简单的故障转移策略
  4. 缓存和书签表示的能力
  5. 选择最适合您的用例的数据格式的灵活性

JAXB

JAXB 提供了一个 API 和工具，用于自动执行 XML 文档和 Java 对象之间的映射。

注释是运行时编组和解组的关键

从现有的 Java 应用程序开始，使用模式生成器创建机器生成的 XML 模式

从现有的 XML 模式开始，使用模式编译器创建机器生成的 Java 应用程序

UDDI

通用描述、发现和集成 (UDDI)

用于通过对丰富的元数据启用强大的查询来提供发布和定位 Web 服务的“元服务”

UDDI 定义了一种机制，以 XML 消息的形式存储已注册 Web 服务的描述

UDDI 规范记录了有关的信息

五个主要元素

  1. businessEntitiy
  2. businessService
  3. bindingTemplate
  4. Technical Models (tModel)（WSDL）
  5. publisherAssertion

JAXR

Java API for XML Registries

JAXR providers for UDDI must be level 0 compliant

JAXR providers for ebXML must be level 1 compliant

适用：

  1. 当您需要扫描大型 XML 文档时
  2. 您需要对解析过程进行更多控制，自己写代码
  3. 您的系统资源有限
  4. 你需要高性能

XSL

将 XML 文档变成其他 XML 文档或其他样式

适用：读取 XML 文档并生成其他文档

SOAP 如何处理二进制数据

传统方法：Base64编码

将二进制数据转换为字符表示形式放入 SOAP 消息中

缺点：消息大小显著增加，编码、解码存在性能开销，效率低下

MTOM

Message Transmission Optimization Mechanism

基于乐观的假设，即发送和接收应用程序实际上不需要二进制值的基于字符的表示

MTOM 使用基于 XOP 的包含，并将二进制数据放在 HTTP 消息的主体中

xop:Include 元素，它通过 href 属性将元素的二进制内容与 SOAP 信息集表示联系起来

基于 URI 的引用

在 SOAP 消息中包含对 Web 资源的基于 URI 的引用

如果接收方希望避免网络流量开销，则可能无法进行包括基于 URI 的引用在内的优化

rep:Representation 元素，它定义了 SOAP 消息信息集中引用的基于 URI 的资源表示如何由已识别的接收者携带和处理

可以与 MTOM XOP 配合使用，优化处理

5

JWS旨在使Java开发人员更容易开发Web服务。因此，它更偏向于“从Java开始”的开发模式。

a) “从Java开始”模式假设您通过开发Java类来创建Web服务。实际上，您只需要在该类上添加@WebService注解，大部分情况下就可以将其部署为Web服务。
JAX-WS和JAXB运行时引擎将使用标准的WSDL/Java和XML/Java映射将该类映射到WSDL文档。

b) 在“从Java开始”开发模式中，如果您需要调整WSDL或XML类型的形状，可以通过向类添加注解来实现。此外，“从Java开始”是WS-Metadata实现所必需的唯一开发模式。

JWS还定义了“从WSDL开始”的开发模式。

c) 使用这种方法，您可以使用WSDL编译器（作为JAX-WS实现的一部分提供）对现有的WSDL进行处理，生成一组实现WSDL的Java程序元素。

d) 使用这种方法，您将获得一组与WSDL相映射的SEI。您的工作是使用适当的业务逻辑来实现SEI，以实现由WSDL定义的Web服务。

\2. 在“从WSDL和Java开始”的开发模式中，您拥有预定义的XML模式类型，必须将其映射到预定义的Java对象（参数和返回类型）。如果没有高度可定制的序列化子系统，这些问题无法优雅地解决。

\3. 因此，几乎任何试图简化序列化问题的Web服务平台（例如通过定义标准序列化上下文，如JAX-RPC 1.1）都会引入在解决“从WSDL和Java开始”问题时需要使用包装代码的需求。

a) 开发人员使用JAX-WS WSDL编译器从现有的WSDL生成SEI，遵循“从WSDL开始”的开发模式，然后使用包装器将该SEI映射到现有的类。

\4. 另一方面，可以认为“从WSDL和Java开始”的开发模式实际上是最容易处理的，因为当WSDL和Java都已定义时，开发人员不需要进行任何设计工作。也就是说，开发人员不需要设计灵活且有用的WSDL或良好的Java API。
从设计师的角度来看，这可能是正确的观点，但从开发人员的角度来看，“从WSDL和Java开始”是使用JWS工具解决最困难的问题。

JWS可以从Java生成有效的WSDL，并且可以从XML Schema编译Java类，但在将现有的Java映射到现有的XML Schema和WSDL方面，它帮助不太多。

6

JWS定义了Port组件（有时也称为Port）作为Web服务的服务器视图。

  • Port组件可以打包为WAR文件（用于作为Servlet端点进行部署）或EJB JAR文件（用于作为EJB端点进行部署）。
  • wsEE是主要标准，定义了部署过程和打包结构（包括部署描述符）。
    WS-Metadata描述了打包类上的注解如何塑造部署（例如，使用WS-Metadata注解可以自定义表示Port组件的WSDL接口）。
    Java EE 5 Port组件的部署对应于WSPA参考系统中的部署子系统的功能。
  1. 此描述跟踪了在步骤1中描述的Port组件部署的Web服务的调用。端点支持用于描述Web服务的WSDL的HTTP GET请求。该WSDL的结构由JAX-WS WSDL/Java映射确定。可以使用JAX-WS、JAXB和WS-Metadata定义的注解来自定义此映射。
  • JAX-WS注解操作所使用的WSDL样式。
  • JAXB注解则操作将Java参数和返回类型表示为XML Schema组件。
  • WS-Metadata注解操作WSDL接口的细节（例如，目标命名空间和WSDL文档中许多XML元素的本地名称）。
    处理WSDL请求不是WSPA的正式部分，但通常是Web服务平台的调用子系统中包含的功能。

  1. 当收到SOAP请求（通过端点的HTTP POST）时，启动已部署的Web服务的调用。

  2. Web服务的端点通常由在部署期间指定的端点URL处侦听的Servlet类实现。此Servlet不由Port组件的开发者指定，而是JWS的内部Java EE 5实现的一部分。在部署过程中，WSEE运行时会读取注解和/或部署描述符，确定端点URL，并配置容器内部以部署侦听器在该端点上。设置端点是WSPA中部署子系统的功能。

  3. 步骤5说明了SOAP请求已成为一组XML参数，这些参数是在WSDL中作为参数指定的XML Schema组件的实例。JAX-WS运行时负责从SOAP消息中提取这些参数。提取XML参数被定义为WSPA中调度过程的一部分，由调用子系统处理。

  1. 客户端调用Web服务的第一步通常涉及使用WSDL到Java映射工具生成服务端点接口（SEI）。SEI提供要调用的Web服务的Java表示。SEI在编译客户端应用程序之前生成，因为其接口定义在客户端应用程序中使用。生成SEI涉及JAXB、JAX-WS和WS-Metadata。
  • 在运行时，使用JAX-WS的javax.xml.ws.Service类的实例来提供要调用的Web服务
